# חלק א' — אפיון פונקציונלי (עברית)

## 1. תמצית הפרויקט

אפליקציית Web (PWA) לראש צוות בברינקס, שמקבלת רשימת יעדים יומית ובונה את סדר הנסיעה האופטימלי — לא "מהקרוב לרחוק" אלא **המסלול שמסיים את כל הסבב (כולל חזרה לסניף) הכי מהר**, תוך עמידה באילוצי זמן, עדיפויות, פקקים בזמן אמת, והפסקות.

**משתמש היעד:** ראש צוות בודד ברכב (עם צוותו). פיילוט עם ראש צוות אחד, ארכיטקטורה מוכנה לריבוי משתמשים ודשבורד ניהולי (שלב 2).

## 2. החלטות אפיון (סיכום התשובות)

| נושא | החלטה |
|---|---|
| היקף סבב | 10–25 יעדים ביום |
| מבנה | ראש צוות נוסע בעצמו, רכב אחד |
| מקור נתונים | מערכת Zebra (אין גישת API כרגע) → קלט גמיש: ידני / קובץ / צילום-מסך עם OCR |
| אילוצי זמן | דד-ליין, "משעה X", חלון X–Y, עדיפות VIP — הכל, בבחירה ידנית פר-יעד |
| התנגשויות | המערכת מתריעה ומציעה פתרונות — ראש הצוות מחליט |
| משך שירות | משתנה פר-לקוח, מוגדר ידנית + למידה מהיסטוריה בעתיד |
| זיהוי הגעה | גיאופנס אוטומטי + אישור ידני ("הגעת? סמן בוצע") |

| דשבורד ניהולי | שלב 2 — אבל כל הנתונים נשמרים בשרת מהיום הראשון |
| הפסקות | הפסקה מוגדרת (משך) + שיבוץ חכם ע"י המערכת |
| גיוון מסלול (אבטחה) | מתג ניתן להפעלה — לברר מול הלקוח |
| שעות | יציאה נבחרת (06:30/07:00/09:00/מותאם), חזרה גמישה עם צפי מחושב |

| יעד | MVP עובד אמיתי לפיילוט |

## 3. זרימת עבודה יומית (User Journey)

**בוקר — תכנון (5 דקות):**
1. ראש הצוות פותח את האפליקציה → מסך "סבב חדש".
2. מזין יעדים באחת מ-3 דרכים: הקלדה ידנית עם השלמת כתובות / העלאת קובץ (Excel/CSV) / צילום מסך של אפליקציית Zebra → AI מחלץ כתובות אוטומטית.
3. לכל יעד: כתובת, שם לקוח, אילוץ זמן (אופציונלי), עדיפות VIP (אופציונלי), משך שירות (ברירת מחדל או מותאם), הערות.
4. בוחר שעת יציאה + משך הפסקת צהריים רצוי.
5. לוחץ "חשב מסלול" → תוך שניות מקבל: סדר יעדים ממוספר על מפה + רשימה, צפי הגעה לכל יעד, שיבוץ הפסקה, צפי חזרה לסניף.
6. אם יש התנגשות אילוצים → באנר אזהרה עם הסבר + 2–3 הצעות פתרון.
7. עריכה ידנית: גרירת סדר, נעילת יעד למיקום, הוספה/הסרה → חישוב מחדש מיידי.
8. "התחל סבב".

**במהלך היום — מצב נסיעה (Live Mode):**
- מסך גדול וברור: היעד הנוכחי, ETA, כפתור ענק "נווט בוויז" (deep link).
- GPS מזהה הגעה (רדיוס ~150 מ') → התראה "הגעת ל[לקוח]? סמן בוצע" → מעבר אוטומטי ליעד הבא.
- ציר התקדמות: בוצע ✓ / נוכחי / הבאים, עם צפי מתעדכן לפי תנועה בפועל.
- סטייה מהצפי (איחור מצטבר) → המערכת מציעה חישוב מחדש.
- הוספת יעד תוך כדי יום → שיבוץ חכם מחדש.
- תזכורת הפסקה בזמן/מקום שנקבע.
- כפתור SOS מהיר (חיוג למוקד).

**סוף יום:**
- הגעה לסניף → סיכום יומי אוטומטי: יעדים שבוצעו, זמנים בפועל מול צפי, ק"מ, זמן שנחסך מול מסלול "נאיבי" (לפי סדר ההזנה) — קלף המכירה שלך מול ברינקס.
- שמירה בהיסטוריה + "שכפל סבב" למחר.

## 4. מנוע האופטימיזציה — הלב של המערכת

- **למה "מהקרוב לרחוק" נכשל:** האלגוריתם הנאיבי (Nearest Neighbor) בוחר בכל צעד את היעד הקרוב ביותר — ואז מגלה בסוף היום שנשאר לו יעד יחיד בקצה השני של המפה, ונוסע 40 דקות סרק. בבעיות כאלה הוא סוטה בממוצע ב-20–25% מהאופטימום. אנחנו לא בונים "סדר לפי מרחק" — אנחנו **פותרים בעיית אופטימיזציה גלובלית**: מתוך כל הסדרים האפשריים (ל-20 יעדים יש 20! ≈ 2.4 קוודריליון אפשרויות), למצוא את זה שמסיים את **כל** היום — כולל החזרה לסניף — הכי מוקדם, בלי להפר אף אילוץ.

- **מודל הבעיה:** VRPTW חד-רכבי (Vehicle Routing Problem with Time Windows) עם depot קבוע להתחלה וסיום, אילוצי חלונות זמן, משכי שירות משתנים, הפסקה, ועדיפויות. נפתר ב-**Google OR-Tools** — פותר תעשייתי חינמי שמשמש חברות לוגיסטיקה גלובליות. ל-25 יעדים: פתרון איכותי תוך 2–5 שניות.

### איך המנוע עובד — 5 שלבים

**שלב 1 — מטריצת נסיעה מודעת-זמן (Time-Dependent Matrix).**
זה ההבדל בין מערכת חובבנית למקצועית: זמן נסיעה מ-א' ל-ב' ב-07:15 שונה לחלוטין מאותה נסיעה ב-11:00. לכן לא בונים מטריצה אחת — בונים **3 מטריצות** לפי חלונות זמן (בוקר-פקקים / אמצע-יום / אחה"צ-פקקים), כל אחת דרך Google Routes API עם `departureTime` מתאים ו-`trafficModel: BEST_GUESS`. הפותר בוחר את המטריצה הרלוונטית לפי השעה החזויה של כל מקטע. עלות: 3 קריאות Matrix בבוקר (זניח), רווח: הערכות שמשקפות פקקים אמיתיים.
→ כל ערך במטריצה מוכפל ב**מקדם הלמידה** הנלמד למקטע (סעיף 4.5), אם קיים.

**שלב 2 — בניית מודל האילוצים.**
| רכיב | מימוש ב-OR-Tools |
|---|---|
| ממד זמן | `AddDimension` עם slack (המתנה מותרת), קיבולת = אורך יום מקסימלי |
| "עד שעה X" (דד-ליין) | `CumulVar(index).SetMax(X)` |
| "משעה X ואילך" | `CumulVar(index).SetMin(X)` — עם slack, כלומר מותר להגיע מוקדם ולהמתין |
| חלון X–Y | `SetRange(X, Y)` |
| משך שירות | נוסף לזמן הנסיעה היוצא מכל צומת (`transit_callback`) |
| חזרה לסניף | הצומת האחרון = depot; **פונקציית המטרה ממזערת את זמן ההגעה אליו** |
| VIP | ראה שלב 3 |
| הפסקה | ראה שלב 4 |

**שלב 3 — פונקציית המטרה: מה בדיוק אנחנו ממזערים.**
```
Minimize:  makespan (שעת החזרה לסניף)
         + Σ penalty_deadline_risk(stop)     # קנס גדל ככל שההגעה קרובה לדד-ליין
         + Σ vip_lateness_weight × arrival_time(vip_stop)
         + drop_penalty × dropped_stops      # רק כשאין פתרון חוקי
```
- **המטרה הראשית היא זמן הסיום, לא סך הק"מ.** זו הגדרה שונה מהותית: לפעמים כדאי לנסוע 5 ק"מ יותר כדי להימנע מ-20 דקות פקק או כדי לתפוס לקוח לפני הדד-ליין שלו.
- **VIP = משקל, לא כלל ברזל:** ה-VIP נדחף להיות מוקדם ככל האפשר, אבל אם הקדמתו תוסיף 40 דקות ליום — המנוע יעדיף להשאיר אותו במקומו הטבעי ולהתריע. משקל ה-VIP חשוף כהגדרה (`vip_weight`) שאפשר לכייל אחרי הפיילוט.
- **חיץ בטיחות:** לכל דד-ליין מוסיפים buffer של 10 דקות (ניתן להגדרה) — כדי שהתוכנית לא תהיה "בדיוק בדקה" ותקרוס מכל עיכוב קטן.

**שלב 4 — ההפסקה כצומת וירטואלי.**
ההפסקה לא נדחפת אחרי החישוב — היא **חלק מהאופטימיזציה עצמה**: צומת וירטואלי עם משך = משך ההפסקה שהוגדר, חלון זמן 11:30–14:00 (ניתן להגדרה), וזמן נסיעה 0 מכל נקודה (כלומר: היא מתרחשת בין שני יעדים, בלי לגרור סטייה). התוצאה: המערכת בוחרת את **הרגע הזול ביותר** להפסקה — למשל בדיוק כשממילא צריך להמתין ללקוח שנפתח ב-13:00. ב-OR-Tools זה `AddDisjunction` על צומת עם חלון זמן, או מנגנון `BreakIntervalVar` המובנה בממד הזמן.

**שלב 5 — פתרון, אימות והסבר.**
- אסטרטגיה: `PATH_CHEAPEST_ARC` לפתרון התחלתי → `GUIDED_LOCAL_SEARCH` (מטא-היוריסטיקה שיודעת "לברוח" מאופטימום מקומי) עם תקציב זמן של 5 שניות.
- **אם אין פתרון חוקי** (התנגשות אילוצים): המנוע לא נכשל בשקט. הוא מריץ שוב עם `AddDisjunction` וקנסות — כלומר "מותר לוותר על יעד בעלות גבוהה" — ומזהה **בדיוק איזה יעד גורם לבעיה**. הממשק מציג: "לא ניתן להספיק את [לקוח א'] עד 10:00 ואת [לקוח ב'] עד 10:30 — הם במרחק 35 דק' זה מזה. אפשרויות: (1) לדחות את ב' ל-11:15 (2) להעביר את א' למחר (3) לוותר על חיץ הבטיחות."
- **שקיפות = אמון:** לכל מסלול המערכת מציגה למה הוא נבחר — "סדר זה חוסך 47 דקות מול הסדר שהוזן" + סימון היעדים שקבעו את המבנה (אילוצי הזמן). ראש צוות שמבין את ההיגיון — סומך על המערכת ולא עוקף אותה.

### חישוב מחדש חכם תוך כדי יום (Re-optimization)
זה מה שהופך את זה מ"מתכנן" ל"שותף":
- יעדים שכבר בוצעו **נעולים כהיסטוריה** — הפותר מקבל את המצב הנוכחי (מיקום GPS, שעה, יעדים שנותרו) ופותר מחדש רק את מה שנשאר.
- טריגרים לחישוב מחדש: הוספת יעד תוך כדי יום, ביטול/דילוג יעד, איחור מצטבר מעל 15 דקות, או בקשה ידנית.
- **אף פעם לא משנה סדר מאחורי הגב:** המערכת מציגה הצעה — "זוהה עיכוב של 22 דק'. סידור מחדש יחסוך 18 דק' ויציל את הדד-ליין של [לקוח]. להחיל?" — וראש הצוות מאשר. יעדים ש"ננעלו" ידנית נשארים במקומם תמיד.

### מצב גיוון (אבטחה)
כשהמתג פעיל: הפותר מייצר 5 פתרונות טובים (`solution_limit`), מסנן את אלה שסוטים ≤7% מהאופטימום, ובוחר אקראית מביניהם — עם העדפה לסדר ששונה מהסדר של אתמול (השוואה מול היסטוריה). התוצאה: אי-חיזוי אמיתי בעלות זניחה של דקות ספורות.

- **Waze:** אין API לאופטימיזציה — משמש רק לניווט בפועל דרך deep link (`waze://?ll=lat,lon&navigate=yes`). כל הלוגיקה על Google Maps Platform.

## 4.5 מנוע הלמידה — המערכת שמשתפרת כל יום 🧠

זה היתרון התחרותי האמיתי: אחרי שבועיים של שימוש, ההערכות של המערכת מדויקות יותר מכל מתחרה, כי היא למדה את המציאות של ברינקס — לא הערכות גנריות של גוגל.

**מה נלמד ואיך:**

1. **משך שירות אמיתי פר-לקוח:**
   - הגיאופנס מתעד אוטומטית: כניסה לרדיוס = `actual_arrival`, יציאה מהרדיוס (או סימון "בוצע" + תזוזה) = `actual_departure`.
   - ההפרש = דגימת משך שירות אמיתית: מהרגע שירדו מהרכב, אספו את הכסף, ועד שהמשיכו ליעד הבא.
   - **אפס עבודה ידנית** — ראש הצוות לא מקליד כלום, המערכת מתעדת לבד.
   - כל לקוח מזוהה ברשומת `customer` קבועה (לפי כתובת מנורמלת + שם) שצוברת דגימות.
   - אחרי 3+ ביקורים: ההערכה עוברת מברירת המחדל לחציון הנלמד (חציון עמיד לחריגים — פעם אחת שהיה בלגן לא הורסת את הממוצע). משקל גבוה יותר לביקורים אחרונים (EWMA).
   - בממשק: ליד לקוח מוכר מופיע תג "⏱ נלמד: 12 דק' (8 ביקורים)" — ראש הצוות רואה שהמערכת מכירה את הלקוחות שלו.

2. **זמני נסיעה אמיתיים בין יעדים:**
   - כל מקטע נסיעה שבוצע נשמר כדגימה: זוג נקודות + שעת יציאה + זמן גוגל החזוי מול הזמן בפועל.
   - המערכת מחשבת **מקדם תיקון** פר-מקטע ופר-חלון שעה (למשל: "בין לקוח א' ללקוח ב' ב-08:00 לוקח בפועל פי 1.15 ממה שגוגל אומר" — כי גוגל לא יודע על החניה, הפריקה, השער האחורי).
   - ETA עתידי = הערכת גוגל × מקדם נלמד (תחום 0.8–1.3 לבטיחות, מופעל רק אחרי 3+ דגימות).

3. **שיפור מתמשך:** כל יום עבודה מזין את המודל. דוח "דיוק חיזוי" שבועי מראה כמה המערכת השתפרה — עוד קלף מכירה מול ברינקס ("המערכת שלכם נהיית חכמה יותר כל יום").

## 4.6 שעון עבודה ולוח בקרה אישי ⏱

ראש הצוות מקבל מעקב שעות מלא — בלי להקליד כלום. המערכת כבר יודעת מתי הוא יצא, איפה היה ומתי חזר.

**איך נמדדות השעות (אוטומטי):**
- **תחילת יום** = יציאה מגיאופנס הסניף (או לחיצה על "התחל סבב" — המוקדם מביניהם).
- **סוף יום** = חזרה לגיאופנס הסניף (או "סיים סבב").
- **הפסקות** מנוכות אוטומטית לפי ההפסקה שנרשמה בפועל (נקודת ההפסקה בסבב + זמן עצירה מזוהה).
- **תיקון ידני זמין תמיד:** אם הגיאופנס פספס (שכח טלפון ברכב, יצא ברגל), ראש הצוות עורך את השעה — כל תיקון מסומן כ"נערך ידנית" בלוג, כך שהנתון נשאר אמין ובר-ביקורת.

**כרטיס "שעון עבודה" בלוח הבקרה:**
| מדד | תצוגה |
|---|---|
| היום | שעון חי שרץ בזמן אמת: `07:12 → כעת · 6:41 שעות` (בפונט Mono, ticker) |
| השבוע | `34:20 / 42:00` — עם טבעת התקדמות זהב |
| החודש | `142:35` + פירוט: רגילות / נוספות |
| מצטבר (מתחילת השימוש) | `487:15 שעות · 62 ימי עבודה` |
| ממוצע יומי | `8:12` — ומגמה מול החודש הקודם |

**פירוט וניתוח:**
- **טבלה יומית:** תאריך | יציאה | חזרה | הפסקה | סה"כ | נוספות | יעדים שבוצעו — עם אפשרות סינון לפי חודש.
- **פילוח זמן היום** (גרף טבעת): נהיגה / שירות אצל לקוחות / המתנה לחלונות זמן / הפסקה. תובנה תפעולית שברינקס תאהב: "23% מהיום זה המתנה — הנה איפה יש כסף על הרצפה".
- **שעות נוספות:** ברירות מחדל ניתנות להגדרה במסך הגדרות (יום תקן, שבוע תקן, סף נוספות). ⚠️ הערכים חייבים להיקבע מול מחלקת השכר של ברינקס וההסכם הקיבוצי החל עליהם — המערכת היא **כלי מעקב ותצוגה, לא מערכת שכר רשמית**. אם יש להם מערכת נוכחות קיימת, בשלב 2 נסנכרן אליה (ולא נתחרה בה).
- **ייצוא:** CSV/PDF של דוח חודשי — כפתור אחד, מוכן לשליחה למנהל.
- **בשלב 2 (דשבורד מנהלים):** אותם נתונים במצטבר לכל הצוותים — מי עמוס, מי מסיים מאוחר באופן קבוע, איפה יש שחיקה.

**ערך מכירה:** זה הופך את האפליקציה מ"עוד כלי ניווט" ל**מערכת התפעול האישית של ראש הצוות** — הוא נכנס אליה גם כדי לראות כמה שעות צבר החודש. שימוש יומיומי = אימוץ = חוזה.

## 8. שדרוגי מומחה — ההצעות שלי מעבר לבריף 💡

אלה הדברים שיהפכו את זה ממוצר טוב למוצר שאין לו מתחרה בישראל. סימנתי מה נכנס ל-MVP (זול ליישום, ערך עצום) ומה משאירים לשלב 2.

### נכנסים ל-MVP

**1. תיקון פין נלמד — "איפה החניון האמיתי" 📍 (הכי חשוב ברשימה)**
גוגל שם את הפין על מרכז הכתובת. הרכב המשוריין חונה בכניסה האחורית, ברחוב הצדדי, ברחבת הפריקה. בכל הגעה, ה-GPS מתעד את **הנקודה שבה הרכב באמת נעצר**. אחרי 3 ביקורים המערכת מזיזה את הפין לחציון הנקודות בפועל, ומציגה "📍 נקודת עצירה מדויקת (נלמדה מ-5 ביקורים)". התוצאה: הניווט בוויז מוביל לנקודה הנכונה, הגיאופנס מדויק, וזמני הנסיעה נכונים. **אף מערכת מדף לא עושה את זה** — וזה מגיע בחינם מנתונים שכבר אוספים. → Phase 4-5

**2. ETA מודע-סיכון (P80) — לא לתכנן על הממוצע ⚖️**
"ממוצע" אומר שב-50% מהמקרים תאחר. ליעדים עם דד-ליין המערכת לא משתמשת בהערכה הממוצעת אלא ב**אחוזון 80** מהדגימות ההיסטוריות — כלומר "בזמן שמכסה 8 מתוך 10 מקרים". יעדים ללא דד-ליין מתוכננים על החציון (יעיל). זו הבחנה שמהנדס אופטימיזציה יעריך, והיא ההבדל בין תוכנית שמחזיקה בשטח לתוכנית שקורסת ב-10:30. → Phase 5

**3. קודי חריגה בעצירה — הזהב התפעולי 🏷️**
בסימון "בוצע", אם היה משהו חריג — לחיצה אחת: `לקוח לא היה מוכן` · `המתנה בשער` · `בעיית חניה` · `סגור` · `עומס בסניף`. שלוש שניות מראש הצוות → נכס אדיר לברינקס: "5 הלקוחות שגוזלים לכם הכי הרבה זמן החודש", "בסניף X ממתינים בממוצע 12 דק' בשער". זה הדוח שמוכר את המערכת להנהלה. גם מזין את הלמידה (דגימה עם חריגה מסומנת ולא מזהמת את החציון). → Phase 4

**4. פילוח לפי יום בשבוע 📅**
זמן שירות בסניף בנק ביום ראשון ≠ יום חמישי. פקקים בבוקר יום א' ≠ יום ו'. הדגימות כבר נשמרות עם חותמת זמן — מוסיפים `day_bucket` (א-ה / ו / חג-ערב חג) ל-`leg_samples` ו-`service_samples`. עלות: שדה אחד. רווח: דיוק משמעותי. → Phase 4

**5. פתרון "היום הראשון" (Cold Start) 🥚**
ביום הראשון אין היסטוריה — ואם ההערכות יהיו גרועות, ראש הצוות יאבד אמון ולא יחזור. פתרון: **קטגוריית לקוח** (סניף בנק / כספומט / רשת קמעונאית / עסק פרטי) עם ערכי ברירת מחדל מכוילים שנקבעים בשיחה אחת עם ראש הצוות המנוסה. אחרי 3 ביקורים הערך הנלמד מחליף. הממשק מציג בכנות: "הערכה: 10 דק' (ברירת מחדל)" מול "12 דק' (נלמד · 8 ביקורים)" — כנות בונה אמון. → Phase 2

**6. תצוגה מקדימה "מה יקרה אם" 🔮**
לפני הוספת יעד תוך כדי יום: "הוספת [לקוח] תוסיף 18 דק' ליום, החזרה לסניף תידחה ל-17:24, והדד-ליין של [לקוח ב'] עדיין מובטח ✓". ראש הצוות מחליט **עם מידע**, לא בניחוש. זו הפיכה של המערכת מכלי לשותף. → Phase 4

**7. עמידות שטח: אופליין + סוללה 🔋**
בלי זה המוצר נופל ביום הראשון:
- **תור פעולות אופליין:** סימון "בוצע", קודי חריגה ותיקוני שעות נשמרים ב-IndexedDB ומסתנכרנים כשהרשת חוזרת. ראש הצוות לא מרגיש כלום.
- **המסלול הפעיל נשמר מקומית** — נפילת רשת לא מוחקת את היום.
- **גיאופנס חסכוני:** `watchPosition` בדיוק גבוה רק כשמתקרבים ליעד (< 1 ק"מ, מחושב מול הפין הבא); אחרת דגימה דלילה. אחרת הסוללה נגמרת ב-13:00 והמוצר מת.
- **Wake Lock** במסך הנסיעה (המסך לא נכבה בזמן נהיגה).
→ Phase 4

**8. מצב הדגמה (Demo Mode) 🎬**
מתג שמפעיל סבב לדוגמה עם 18 לקוחות בגוש דן ונתוני היסטוריה מזויפים — כדי שתוכל להדגים למנהלים בברינקס את **המוצר המלא** בלי לחכות לנתונים אמיתיים, ובלי לחשוף לקוחות אמיתיים. כלי המכירה הכי חשוב שלך. → Phase 5

### שלב 2 (אחרי שהפיילוט מוכיח את עצמו)

- **זיהוי אנומליות תפעוליות:** עצירה שנמשכת פי 2.5 מהחציון הנלמד → התראה לדשבורד המנהלים ("בדוק מה קורה"). בחברת שינוע כספים לחריגה כזו יש משמעות תפעולית — אבל **זה לא תחליף למערכת אבטחה**, ויש להציג אותו ככלי תפעולי בלבד ולתאם עם קצין הביטחון שלהם.
- **מקדם מזג אוויר:** גשם כבד מוסיף 10–20% לזמני נסיעה. Weather API + מקדם על המטריצה.
- **דחיית יעד למחר:** "אין זמן היום" → היעד עובר לתכנון של מחר עם דגל עדיפות.
- **דוח ROI לברינקס:** אגרגציה של כל הצוותים — "החודש נחסכו 240 שעות עבודה". זה המסמך שמצדיק את החוזה השנתי.
- **אופטימיזציה רב-רכבית:** אם ברינקס ירצו לחלק יעדים בין כמה צוותים — OR-Tools כבר תומך, המודל בנוי לזה.
- **התאמת קטגוריות אוטומטית** מנתוני החריגות.

## 10. הקמת שירותים ומפתחות (לפני Phase 2)

**Google Cloud Platform:**
1. פרויקט חדש → הפעלת חיוב (יש 300$ קרדיט התחלתי + ~200$/חודש חינם ב-Maps).
2. הפעלת 4 APIs: **Routes API** (מטריצה + ניווט), **Geocoding API**, **Places API (New)** (השלמת כתובות), **Maps JavaScript API** (תצוגה).
3. שני מפתחות נפרדים:
   - `GOOGLE_MAPS_SERVER_KEY` — ללא הגבלת referrer, מוגבל ל-APIs של השרת, נשמר רק ב-`.env` של הבקאנד.
   - `GOOGLE_MAPS_BROWSER_KEY` — מוגבל ל-HTTP referrer של הדומיין שלך + Maps JS API בלבד.
4. הגדרת **מכסות יומיות** (Quotas) — הגנה מפני הפתעת חיוב.

**Anthropic:** מפתח API ל-OCR (`ANTHROPIC_API_KEY`) — בשרת בלבד. עלות: אגורות לצילום.

⚠️ **כלל ברזל:** אף מפתח לא נחשף לפרונטאנד מלבד ה-Browser Key המוגבל. כל קריאה אחרת עוברת דרך פרוקסי ב-FastAPI.

## 11. פריסה לפיילוט (Deployment)

- **HTTPS הוא חובה, לא המלצה** — Geolocation API ו-Service Worker לא עובדים ב-HTTP. בלי זה אין מוצר.
- **המלצה לפיילוט:** Railway או Render — בקאנד (Docker), Postgres מנוהל עם גיבויים, פרונט כסטטי + CDN, HTTPS אוטומטי, פריסה מ-Git. עלות: ~20-40$/חודש.
- **גיבויים:** יומי אוטומטי + בדיקת שחזור לפני תחילת הפיילוט. אתה מחזיק נתוני לקוחות של ברינקס — אובדן נתונים = סוף הקשר.
- **ניטור:** Sentry (שגיאות) + uptime monitor. אם המערכת נופלת ב-07:00 ואתה מגלה ב-11:00 — נגמר הפיילוט.
- ⚠️ **מיקום הנתונים (Data Residency):** ברינקס עשויה לדרוש שמידע על לקוחותיה יישמר בישראל/באיחוד האירופי ולא בשרתים אמריקאיים. **לברר לפני בחירת ספק הענן** — זה יכול לשנות את כל הארכיטקטורה בדיעבד. שאלה לפגישה.
- **התקנה על מכשיר:** PWA → "הוסף למסך הבית" באנדרואיד/כרום. שווה להכין דף הוראות עברית של 3 שלבים לראש הצוות.

## 14. Phase 0 — לאמת לפני שבונים 🚨

**זה הסעיף הכי חשוב במסמך. הוא נכתב אחרון והוא צריך לקרות ראשון.**

כל המסמך הזה עומד על **הנחה אחת לא בדוקה**: שראש הצוות מסדר את הסבב שלו בצורה לא אופטימלית, ושאלגוריתם יעשה זאת טוב יותר במידה משמעותית. אם ההנחה נכונה — יש לך עסק. אם ראש צוות ותיק עם 15 שנות ניסיון כבר מסדר את היום שלו טוב ב-3% מהאופטימום — **אין מוצר**, ואת זה עדיף לגלות ביום אחד ולא בחודש הרביעי.

### שלוש בדיקות. שבוע אחד. לפני Phase 1.

## 15. הצד המסחרי — להחליט לפני הקוד ⚖️

זה לא נושא טכני, אבל זו ההחלטה עם ההשלכה הכספית הגדולה ביותר בפרויקט — והיא נסגרת **לפני** הפיתוח, לא אחריו.

**מי מחזיק בקניין הרוחני?**
- אם תבנה בתור **work-for-hire** — הקוד שייך לברינקס. קיבלת תשלום פרויקט חד-פעמי, ולא תוכל למכור את אותה מערכת לאיילון, לדואר, לחברות שינוע אחרות, או לכל חברת לוגיסטיקה בעולם.
- אם תשמור על ה-IP ותיתן **רישיון שימוש** — יש לך מוצר. ברינקס היא לקוח ראשון, לא בעלים. אותה מערכת נמכרת שוב ושוב, וההכנסה חוזרת.

**ההפרש בין שתי המילים האלה בחוזה הוא פי 10-50 בשווי לאורך זמן.** חברות גדולות ידרשו בעלות כברירת מחדל — זה סטנדרטי אצלן, ואם לא תעלה את הנושא, הוא ייסגר לרעתך בשקט.

⚠️ **אני לא עורך דין, וזו לא ייעוץ משפטי.** אבל אני יכול להגיד לך בביטחון: **אל תחתום על שום דבר מול ארגון בסדר גודל כזה בלי עורך דין שמתמחה בהסכמי תוכנה.** העלות של שעתיים ייעוץ היא אחוזים בודדים מהסיכון. נושאים לסגור: בעלות IP, רישיון מול העברה, הסכם עיבוד נתונים (אתה מחזיק מידע על לקוחות ברינקס), אחריות ושיפוי, ותנאי הפיילוט (בתשלום או חינם? מה קורה בסופו?).

**עוד שתי נקודות לפני הפיילוט:**
- **מי מחליט בברינקס?** מנהל סניף? מנהל תפעול אזורי? הנהלה? אם אתה מדבר עם מי שאין לו תקציב — אתה בונה עבור אוהד, לא עבור קונה. מפה את שרשרת ההחלטה לפני שאתה משקיע חודשים.
- **תמיכה:** אם המערכת נופלת ב-06:30 והסבב לא יוצא — מי עונה? צריך התחייבות מוגדרת, ולו בסיסית, לפני שראש צוות תלוי בזה כדי לעשות את העבודה שלו.

# חלק ב' — מערכת העיצוב "Vault" (Premium Security)

לא "עוד אפליקציה כהה עם זהב" — זהות חזותית שנבנתה מהעולם של ברינקס עצמו: כספות, שטרות, חותמות אבטחה, פלדה משוריינת. כל אלמנט עיצובי מגיע משם.

## הרעיון המרכזי + אלמנט החתימה
- **חתימת המוצר: "חוגת הכספת" (Vault Dial)** — טבעת התקדמות מעגלית בראש המסך, מעוצבת כחוגה של כספת: שנתות עדינות מסביב, מחט זהב, ובמרכזה מספר היעדים שהושלמו מתוך הסך. עם כל יעד שמסומן "בוצע" — החוגה **מסתובבת בנקישה** (אנימציית rotate קצרה עם easing מכני + רטט הפטי קל במובייל) כמו נעילת בריח. בסוף היום, כשהסבב הושלם — החוגה "ננעלת" עם אנימציית סגירה וחותם זהב "הסבב הושלם". זה הרגע שראש הצוות מחכה לו כל יום, וזה מה שברינקס יזכרו מהדמו.
- **טקסטורת רקע Guilloché** — הדוגמה הגיליוכית (הקווים המסולסלים העדינים שמודפסים על שטרות כסף) כטקסטורת SVG עדינה מאוד (אטימות 3–4%) על רקעי המסכים הראשיים. רמז תת-סיפי מהעולם של שטרות — אף מתחרה לא ייראה ככה.
- **זהב מטאלי, לא זהב שטוח:** כל אלמנט זהב (כפתור ראשי, מחט החוגה, תגי VIP) מקבל gradient מטאלי עדין (`linear-gradient(135deg, #E8CE7A, #C9A227 55%, #9C7A14)`) + הדגשת קצה עליון בהירה — תחושת מתכת אמיתית, לא צבע.

- **Dark-first** (נהיגה, שמש, לילה) עם מתג Light mode (גווני שנהב-פלדה, אותה שפה).
- רקע ראשי: gradient עומק `radial-gradient` מ-navy-900 במרכז ל-navy-950 בקצוות + guilloché — תחושת חלל, לא משטח שטוח.
- זהב = אקשן ראשי + חתימה בלבד. משמעת: מסך ממוצע מכיל אלמנט זהב אחד דומיננטי.

## טיפוגרפיה (זוג פונטים מכוון, לא ברירת מחדל)
- **תצוגה/כותרות: Suez One** — סלאב-סריף עברי כבד וסמכותי, מריח כמו שלט של בנק ותיק. לכותרות מסך, מספר היעד הנוכחי, והחוגה. שימוש במשורה — הוא הקול, לא הרעש.
- **גוף: Heebo** 400/500/700 — קריא, נקי, עברי מצוין.
- **דאטה/שעון: IBM Plex Mono** (או Space Grotesk) עם `font-variant-numeric: tabular-nums` — כל ETA, שעה, ק"מ ומונה. מספרים מיושרים שלא "קופצים" כשמתעדכנים.
- סקאלה: Display 34/Suez, H1 26/Suez, H2 20/Heebo 700, Body 16/Heebo 400, Data 15/Mono, Caption 13/Heebo 500.
- **Live Mode:** שם יעד נוכחי 34/Suez, ETA 44/Mono 700 — נקרא במבט של חצי שנייה ממרחק זרוע.
- מספרים ושעות תמיד LTR בתוך טקסט RTL (`unicode-bidi`).

## אפקטים חזותיים (מדויקים, לא ראוותניים)
- **כרטיסי יעד "לוחית פלדה":** רקע `--surface-raised` עם gradient אנכי עדין (1.5% הבהרה למעלה = תאורה מלמעלה), border 1px, צל רך כפול (ambient + key). פס סטטוס בצד: gradient זהב מטאלי=VIP, כחול=רגיל, ירוק=בוצע, אדום פועם עדין=בסיכון דד-ליין.
- **תג "נלמד" 🧠:** לקוח עם נתוני היסטוריה מקבל תג קטן עם קו-מתאר זהב: "⏱ 12 דק' · 8 ביקורים" — הדרך שבה החוכמה של המערכת נראית לעין.
- **מיקרו-אינטראקציות:** כפתור ראשי — scale(0.97) בלחיצה + glow זהב מתרחב; סימון "בוצע" — צ'קמארק מצויר ב-SVG stroke animation (240ms) ואז הכרטיס מתקפל לרשימת ההושלמו; ETA שמתעדכן — ticker ספרות (הספרה הישנה מחליקה למעלה, החדשה נכנסת מלמטה); הוספת יעד — slide-in עם stagger של 40ms בין כרטיסים.
- **מפה בהתאמה:** Google Maps עם JSON styling כהה מותאם לפלטה (כבישים navy-700, מים navy-950); קו המסלול gradient זהב→ענבר; מרקרים ממוספרים כעיגולי פלדה עם ספרת Mono; המרקר הנוכחי — טבעת זהב עם pulse איטי (2.4s).
- **חוקי משמעת:** אנימציות 160–280ms בלבד, easing `cubic-bezier(0.2, 0.8, 0.2, 1)`; אפס אנימציות דקורטיביות במסכי נסיעה (בטיחות!); כיבוד מלא של `prefers-reduced-motion`.

## ביצועים ומהירות (פרימיום = מהיר)
- טעינה ראשונה < 2 שניות ב-4G: code-splitting לפי מסך, פונטים עם `font-display: swap` + preload, אייקונים כ-SVG sprite (לא ספריית ענק).
- **Optimistic UI:** סימון "בוצע", גרירת סדר, עריכת יעד — מגיבים מיידית בממשק, הסנכרון לשרת ברקע. אפס המתנות מיותרות.
- Skeleton loaders בצורת התוכן האמיתי (לא ספינרים) בכל טעינת נתונים; חישוב מסלול מציג התקדמות חכמה ("מחשב מרחקים… מסדר יעדים… בודק אילוצים…").
- Service Worker: העמוד והנכסים נטענים מקאש — פתיחה שנייה מיידית; המסלול האחרון זמין גם בנפילת רשת רגעית.

## עקרונות UX קריטיים
- **In-vehicle first:** מטרות מגע ≥ 56px ב-Live Mode (כפפות/טלטולים), ניגודיות AA מינימום ו-AAA לטקסט קריטי (ETA, שם יעד), כל פעולה נפוצה בלחיצה אחת מהמסך הראשי.
- RTL מלא (dir="rtl") בכל רכיב, כולל גרירה, מגירות ומפה.
- מצבי מערכת מלאים בעברית: loading (skeleton), empty state עם איור וקריאה לפעולה ("אין עדיין יעדים — הוסף את הראשון"), שגיאות שמסבירות מה קרה ומה עושים (לא "אירעה שגיאה").
- נגישות: פוקוס מקלדת נראה (טבעת זהב), aria-labels בעברית, ניווט מקלדת מלא במסכי התכנון.
